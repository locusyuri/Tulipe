# 接口设计说明：连接测试与 SQL 执行

> 适用范围：Tulipe 后端（Spring Boot + Kotlin）。
> 本文档记录连接测试、结构化 SQL 接口、兼容原始 SQL 接口的设计与状态流转。

---

## 1. 接口定位

当前后端围绕 6 个核心接口 + 1 个兼容接口：

1. `POST /data-source/connect`
   - 用途：测试连接并持久化连接信息。
   - 返回：`connectionId`。

2. `POST /sql/query`
   - 用途：结构化查询。
   - 入参：`StructuredQueryRequest`。

3. `POST /sql/insert`
   - 用途：结构化插入。
   - 入参：`StructuredInsertRequest`。

4. `POST /sql/update`
   - 用途：结构化更新。
   - 入参：`StructuredUpdateRequest`。

5. `POST /sql/delete`
   - 用途：结构化删除。
   - 入参：`StructuredDeleteRequest`。

6. `POST /sql/ddl`
   - 用途：结构化 DDL。
   - 当前支持：`CREATE / DROP / TRUNCATE`。

7. `POST /sql/execute`（兼容）
   - 用途：执行原始 SQL（`connectionId + sql`）。
   - 说明：用于复杂语句和历史前端兼容。

---

## 2. 分层职责

- `DataSourceController`
  - 只负责连接测试：`/data-source/connect`。
- `SqlController`
  - 负责结构化 SQL 接口和兼容执行接口。
- `DataSourceService`
  - 负责参数校验、SQL 构建、执行。
- `ConnectionPoolManager`
  - 负责按连接配置复用连接池。

---

## 3. 数据模型

### 3.1 连接模型

连接信息使用 `Connection`（含多态 `config`）做测试连接输入；
持久化时映射到 `connection_profile` 表。

### 3.2 结构化 SQL 请求

- `StructuredQueryRequest`
  - `connectionId`、`table`、`columns`、`filters`、`orderBy`、`page`、`pageSize`
- `StructuredInsertRequest`
  - `connectionId`、`table`、`values`
- `StructuredUpdateRequest`
  - `connectionId`、`table`、`setValues`、`filters`
- `StructuredDeleteRequest`
  - `connectionId`、`table`、`filters`
- `StructuredDdlRequest`
  - `connectionId`、`action`、`table`、`columns`

### 3.3 兼容请求

- `ExecuteSqlRequest`
  - `connectionId`、`sql`

## 4. 状态流转

### 4.1 连接测试 `/data-source/connect`

```text
前端提交 Connection
  ↓
Controller 接收请求
  ↓
Service 校验 dbType / config.type
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行 SELECT 1 探活
  ↓
ConnectionProfileRepository 保存到 SQLite
  ↓
返回 connectionId
```

### 4.2 结构化查询 `/sql/query`

```text
前端提交 StructuredQueryRequest
  ↓
Service 校验表名/字段名/过滤条件/分页参数
  ↓
按 connectionId 加载连接配置
  ↓
后端构建参数化 SELECT SQL
  ↓
执行 queryForList
  ↓
返回 SqlQueryResult
```

### 4.3 结构化写入 `/sql/insert|update|delete`

```text
前端提交结构化请求
  ↓
Service 校验 table / values / filters
  ↓
按 connectionId 加载连接配置
  ↓
后端构建参数化 INSERT/UPDATE/DELETE
  ↓
执行 update
  ↓
返回 SqlMutationResult
```

说明：
- `update` 默认要求 `filters` 非空，避免误更新全表。
- `delete` 默认要求 `filters` 非空，避免误删全表。

### 4.4 结构化 DDL `/sql/ddl`

```text
前端提交 StructuredDdlRequest
  ↓
Service 校验 action / table / columns
  ↓
按 connectionId 加载连接配置
  ↓
后端构建 CREATE/DROP/TRUNCATE
  ↓
执行 update
  ↓
返回 SqlMutationResult
```

### 4.5 兼容执行 `/sql/execute`

```text
前端提交 ExecuteSqlRequest(connectionId + sql)
  ↓
Service 识别 SQL 类型
  ↓
执行查询或写入
  ↓
返回 SqlExecutionResult
```

---

## 5. 返回结果

- `/data-source/connect`
  - `TulipeResult<Long>`，`data = connectionId`
- `/sql/query`
  - `TulipeResult<SqlQueryResult>`（`rows`）
- `/sql/insert|update|delete|ddl`
  - `TulipeResult<SqlMutationResult>`（`affectedRows`）
- `/sql/execute`
  - `TulipeResult<SqlExecutionResult>`（兼容）

---

## 6. 当前支持与限制

### 6.1 当前支持

- MySQL、PostgreSQL（JDBC）
- 连接池复用
- 结构化：`SELECT / INSERT / UPDATE / DELETE / CREATE / DROP / TRUNCATE`
- 原始 SQL 兼容执行（`/sql/execute`）

### 6.2 当前限制

- 非 JDBC 数据源暂未接入（Redis/图数据库/时序库）
- 结构化 DDL 首版不支持 `ALTER`
- 查询结果暂未做大结果裁剪
- 连接更新/删除后的连接池驱逐策略待补齐

---

## 7. 为什么采用结构化输入

- 前端不用手拼 SQL，使用门槛更低
- 后端统一参数化执行，降低注入风险
- 更容易做字段级权限与审计
- 与未来多数据源（非 SQL）能力更兼容
