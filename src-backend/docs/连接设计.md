# 接口设计说明：连接测试与 SQL 执行

> 适用范围：Tulipe 的后端（Spring Boot + Kotlin）部分。当前文档说明**连接测试**、**查询**、**更新**、**删除**以及兼容执行入口的设计、状态流转和职责边界，便于后续扩展 SQLite 持久化、连接列表、连接编辑等能力。

---

## 1. 接口定位

当前后端围绕五个核心接口 + 一个兼容接口展开：

1. `POST /data-source/connect`
   - 用途：测试一个连接配置是否可用。
   - 特点：连接测试成功后会把连接信息保存到 SQLite，并返回 `connectionId`。
   - 典型场景：用户在前端填完连接信息后，点击“测试连接”。

2. `POST /sql/query`
   - 用途：执行查询语句。
   - 特点：当前请求体携带 `connectionId + sql`；只返回 `rows`。
   - 典型场景：用户在 SQL 编辑器里执行 `SELECT`。

3. `POST /sql/update`
   - 用途：执行写入类语句（当前首版覆盖 `INSERT / UPDATE`）。
   - 特点：当前请求体携带 `connectionId + sql`；只返回 `affectedRows`。
   - 典型场景：用户在 SQL 编辑器里执行插入或更新语句。

4. `POST /sql/delete`
   - 用途：执行删除语句。
   - 特点：当前请求体携带 `connectionId + sql`；只返回 `affectedRows`。
   - 典型场景：用户在 SQL 编辑器里执行 `DELETE`。

5. `POST /sql/ddl`
   - 用途：执行 DDL 语句（`CREATE / ALTER / DROP / TRUNCATE`）。
   - 特点：当前请求体携带 `connectionId + sql`；返回 `affectedRows`（多数数据库 DDL 通常为 0）。
   - 典型场景：建表、改表、删表等结构变更。

6. `POST /sql/execute`
   - 用途：旧的统一执行入口，当前保留为兼容接口。
   - 特点：仍可根据 SQL 类型执行查询或写入，但新前端应迁移到 query/update/delete/ddl。
   - 典型场景：历史版本前端或临时回滚场景。

---

## 2. 总体设计思路

### 2.1 分层职责

当前采用比较清晰的三层职责：

- **Controller 层**
  - 只负责接收请求、调用 Service、返回统一结果。
  - 不写数据库连接细节，不直接拼 SQL。
  - 当前按职责拆分为 `DataSourceController`（连接测试）与 `SqlController`（SQL 查询/更新/删除）。

- **Service 层**
  - 负责业务判断与流程控制。
  - 决定是测试连接还是执行 SQL。
  - 遇到错误直接抛出，由全局异常处理器统一返回失败结果。

- **ConnectionPoolManager 层**
  - 负责按连接配置创建和缓存连接池。
  - 相同连接配置命中同一个池，避免重复建连。
  - 程序退出时统一关闭所有池。

### 2.2 设计原则

- **统一入口**：前端只需要知道拆分后的 query / update / delete / ddl 接口，以及兼容入口。
- **统一错误处理**：失败统一交给全局异常处理器。
- **资源复用**：外部数据库连接通过连接池复用，不长期持有单个 JDBC Connection。
- **可扩展**：后续可继续增加 Redis、MongoDB、图数据库等非 JDBC 类型。

---

## 3. 数据模型概览

### 3.1 连接请求模型

当前连接信息采用嵌套结构：

- 外层是通用字段：
  - `name`：用户自定义连接名称
  - `group`：分组信息
  - `dbType`：数据库类型
  - `config`：具体数据库配置对象

- 内层是多态配置对象：
  - `MysqlConnectionConfig`
  - `PostgreSqlConnectionConfig`

### 3.2 为什么这样设计

因为不同数据库的连接信息不完全一样：

- MySQL 这类 JDBC 数据库：通常只需要 `host / port / username / password`
- PostgreSQL：除了基础信息，还需要 `database`

所以把通用字段与数据库专属字段拆开，可以避免“所有数据库都强制带同一个字段”的问题。

## 4. 接口状态流转

## 4.1 连接测试 `/data-source/connect`

这个接口只做一件事：**验证连接是否可用**。

### 状态流转

```text
前端提交连接配置
  ↓
Controller 接收请求
  ↓
Service 校验 dbType / config.type
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行最小探活 SQL（SELECT 1）
  ↓
ConnectionProfileRepository 保存连接信息到 SQLite
  ↓
成功：返回 TulipeResult.success(connectionId)
失败：抛出异常 → 全局异常处理器统一返回失败结果
```

### 状态解释

- **接收状态**：请求进入后端。
- **校验状态**：检查请求结构是否匹配当前支持的数据库类型。
- **建池/取池状态**：从缓存中复用已存在池，若不存在则创建。
- **探活状态**：执行 `SELECT 1` 验证数据库是否可连通。
- **保存状态**：探活成功后保存连接配置，生成 `connectionId`。
- **结束状态**：
  - 成功：返回 `connectionId`。
  - 失败：统一输出失败信息。

### 成功后的含义

“连接成功”表示：

- 当前连接信息可用
- 认证通过
- 网络可达
- 数据库服务响应正常

它**不代表**后续 SQL 一定执行成功，因为后续语句还会受权限、表结构、SQL 内容影响。

---

## 4.2 查询 `/sql/query`

这个接口只做一件事：**执行查询类 SQL，并返回 rows**。

### 状态流转

```text
前端提交 connectionId + SQL
  ↓
Controller 接收请求
  ↓
Service 校验 SQL 非空
  ↓
Service 按 connectionId 读取 SQLite 中的连接配置
  ↓
Service 校验 SQL 类型为 SELECT
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行查询
  ↓
返回 SqlQueryResult
  ↓
失败：抛出异常 → 全局异常处理器统一返回失败结果
```

### 状态解释

- **查询校验状态**：只允许 `SELECT`。
- **执行状态**：通过 `JdbcTemplate.queryForList(...)` 返回结果集。
- **结束状态**：返回 `rows`，不再混入影响行数字段。

## 4.3 更新 `/sql/update`

这个接口用于执行写入类 SQL，当前首版覆盖 `INSERT / UPDATE`。

### 状态流转

```text
前端提交 connectionId + SQL
  ↓
Controller 接收请求
  ↓
Service 校验 SQL 非空
  ↓
Service 按 connectionId 读取 SQLite 中的连接配置
  ↓
Service 校验 SQL 类型为 INSERT / UPDATE
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行写入
  ↓
返回 SqlMutationResult
  ↓
失败：抛出异常 → 全局异常处理器统一返回失败结果
```

### 状态解释

- **更新校验状态**：只允许 `INSERT / UPDATE`。
- **执行状态**：通过 `JdbcTemplate.update(...)` 返回影响行数。
- **结束状态**：返回 `affectedRows`，不混入查询结果集。

## 4.4 删除 `/sql/delete`

这个接口只做一件事：**执行删除类 SQL，并返回影响行数**。

### 状态流转

```text
前端提交 connectionId + SQL
  ↓
Controller 接收请求
  ↓
Service 校验 SQL 非空
  ↓
Service 按 connectionId 读取 SQLite 中的连接配置
  ↓
Service 校验 SQL 类型为 DELETE
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行删除
  ↓
返回 SqlMutationResult
  ↓
失败：抛出异常 → 全局异常处理器统一返回失败结果
```

### 状态解释

- **删除校验状态**：只允许 `DELETE`。
- **执行状态**：通过 `JdbcTemplate.update(...)` 返回影响行数。
- **结束状态**：返回 `affectedRows`。

## 4.5 DDL `/sql/ddl`

这个接口用于执行结构定义类 SQL（DDL）。

- 允许类型：`CREATE / ALTER / DROP / TRUNCATE`
- 返回结构：`SqlMutationResult(sqlType, affectedRows)`
- 说明：很多数据库执行 DDL 时 `affectedRows` 会是 `0`，这是正常现象。

## 4.6 兼容接口 `/sql/execute`

这个接口是旧的统一执行入口，当前仅作为兼容保留：

- 如果前端还没迁移，可以继续临时调用。
- 新前端应优先使用 `query / update / delete / ddl`。
- 后续等所有调用方迁移完成后，可以再考虑删除。

## 5. 连接池使用方式

### 5.1 连接池的目标

连接池不是“永远保持一个连接不放”，而是：

- 保持一个可复用的数据源对象
- 每次真正执行 SQL 时，从池里借出一个连接
- 用完后由连接池自动归还

### 5.2 当前实现方式

当前 `ConnectionPoolManager` 的行为是：

1. 根据连接配置生成一个缓存 key
2. 如果该 key 对应的池已存在，直接复用
3. 如果不存在，则创建新的 `HikariDataSource`
4. 程序退出时统一关闭所有池

### 5.3 为什么要这么做

好处有三个：

- **性能更好**：避免每次请求都重新建连
- **资源更稳**：连接会自动复用和回收
- **适合多连接场景**：每个用户连接都可以有自己的池

## 6. 返回结果说明

### 6.1 连接测试返回

`POST /data-source/connect`

- 成功：
  - 统一返回 `TulipeResult.success(connectionId)`
  - `data = connectionId`
- 失败：
  - 统一由异常处理器返回失败结果

### 6.2 查询返回

`POST /sql/query`

成功时外层为 `TulipeResult.success(...)`，其中 `data` 是 `SqlQueryResult`：

- `sqlType`
  - 固定为 `SELECT`
- `rows`
  - 查询结果集

### 6.3 更新返回

`POST /sql/update`

成功时外层为 `TulipeResult.success(...)`，其中 `data` 是 `SqlMutationResult`：

- `sqlType`
  - `INSERT` 或 `UPDATE`
- `affectedRows`
  - 影响行数

### 6.4 删除返回

`POST /sql/delete`

成功时外层为 `TulipeResult.success(...)`，其中 `data` 是 `SqlMutationResult`：

- `sqlType`
  - 固定为 `DELETE`
- `affectedRows`
  - 影响行数

### 6.5 DDL 返回

`POST /sql/ddl`

成功时外层为 `TulipeResult.success(...)`，其中 `data` 是 `SqlMutationResult`：

- `sqlType`
  - 固定为 `CREATE` / `ALTER` / `DROP` / `TRUNCATE`
- `affectedRows`
  - 影响行数（多数数据库 DDL 为 `0`）

### 6.6 兼容执行返回

`POST /sql/execute`

兼容接口仍返回 `SqlExecutionResult`：

- `sqlType`
  - SQL 类型
- `rows`
  - 仅查询语句使用
- `affectedRows`
  - 仅写入语句使用

### 6.7 这样设计的好处

- 接口返回结构统一
- 前端判断简单
- 后续可以继续扩展字段，比如：
  - `columns`
  - `costMs`
  - `truncated`
  - `warningMessage`

## 7. 当前支持范围与限制

### 7.1 当前支持

目前首版已支持：

- MySQL
- PostgreSQL
- JDBC 类型的连接池管理
- `SELECT / INSERT / UPDATE / DELETE / CREATE / ALTER / DROP / TRUNCATE` 的 SQL 执行

### 7.2 当前限制

- 非 JDBC 类型数据库暂未接入
  - 例如 Redis、图数据库、时序数据库等
- SQL 执行已拆分为 query / update / delete / ddl 四类接口
- 当前 `/sql/*` 仍是原始 SQL 入参模式（`connectionId + sql`）
- 面向普通业务页面的结构化接口（不直传 SQL）尚未落地
- 结果集暂未做分页与大结果裁剪
- 连接配置更新/删除后的池驱逐逻辑后续再补
- 旧的 `/sql/execute` 仍保留为兼容入口

---

## 8. 异常处理原则

当前采用统一异常处理策略：

- Service 层发现错误后直接抛出异常
- Controller 不吞异常
- 全局异常处理器负责转换成统一 `TulipeResult.failure(...)`

这样可以保证：

- 错误口径统一
- 前端处理简单
- 业务代码不被大量 `try/catch` 污染

---

## 9. 后续扩展建议

后续如果继续增强，建议按以下顺序推进：

1. 加上连接列表、编辑、删除接口
2. 完善连接配置更新/删除时的连接池驱逐
3. 为 query / update / delete 抽象更细的执行器模板
4. 增加查询结果分页与大结果裁剪
5. 接入 Redis、图数据库、时序数据库等非 JDBC 类型

## 11. 当前新增的连接保存流转

现在 `/data-source/connect` 的流程已经变成：

```text
前端提交连接配置
  ↓
Controller 接收请求
  ↓
Service 校验 dbType / config.type
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行最小探活 SQL（SELECT 1）
  ↓
连接测试通过
  ↓
ConnectionProfileRepository 保存连接信息到 SQLite
  ↓
返回 connectionId
```

这一步的意义是：

- 前端不需要自己维护一大段连接配置
- 后端可以用 `connectionId` 统一管理连接信息
- 后续新增“按连接 ID 执行 SQL”时，链路会更自然

## 12. 一句话总结

当前这套设计的核心目标是：

> **通过统一接口 + 连接池复用 + 服务层分流，把“测试连接”和“执行 SQL”做成可扩展、易理解、便于后续接 SQLite 持久化的基础能力。**

## 13. 结构化接口规划（前端不直传 SQL）

本节用于回答一个关键设计问题：

> 是否只能让前端传原始 SQL？

答案是：**不是**。当前 `/sql/*` 主要服务 SQL 控制台和高级用户；后续会补充结构化接口，给普通业务页面使用。

### 13.1 双轨模式

1. **原始 SQL 模式（已实现）**
   - 路径：`/sql/query`、`/sql/update`、`/sql/delete`、`/sql/ddl`
   - 入参：`connectionId + sql`
   - 适合：SQL 编辑器、调试、DBA 操作

2. **结构化模式（规划中）**
   - 路径建议：`/meta/*`、`/records/*`
   - 入参：结构化字段（表名、条件、分页、更新值等），由后端生成 SQL 并参数化执行
   - 适合：普通页面 CRUD、低代码场景、可控权限场景

### 13.2 规划接口（建议）

- 元数据：
  - `GET /meta/tables?connectionId=...`
  - `GET /meta/columns?connectionId=...&table=...`
- 查询：
  - `POST /records/search`
  - `POST /records/count`
- 写入：
  - `POST /records/insert`
  - `POST /records/update`
  - `POST /records/delete`

### 13.3 DTO 方向（建议字段）

- `RecordSearchRequest`
  - `connectionId`、`table`、`columns`
  - `filters`（字段、操作符、值）
  - `sort`、`page`、`pageSize`
- `RecordInsertRequest`
  - `connectionId`、`table`、`values`
- `RecordUpdateRequest`
  - `connectionId`、`table`、`setValues`、`filters`
- `RecordDeleteRequest`
  - `connectionId`、`table`、`filters`

### 13.4 为什么要做结构化模式

- 前端不需要拼接 SQL，使用门槛更低
- 后端可统一参数化处理，降低注入风险
- 更容易做字段级权限、审计和灰度策略
- 更适合后续扩展到 Redis/图数据库等“非 SQL”数据源

### 13.5 落地阶段建议

1. **Phase 1（并行）**：保留 `/sql/*`，新增 `/meta/*` + `/records/search`
2. **Phase 2（迁移）**：业务页面默认走结构化接口，仅 SQL 控制台继续走 `/sql/*`
3. **Phase 3（收敛）**：将 `/sql/*` 归类为高级能力，按角色限制
