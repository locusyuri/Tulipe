# 接口设计说明：连接测试与 SQL 执行

> 适用范围：Tulipe 的后端（Spring Boot + Kotlin）部分。当前文档只说明**连接测试**与**SQL 执行**两个相关接口的设计、状态流转和职责边界，便于后续扩展 SQLite 持久化、连接列表、连接编辑等能力。

---

## 1. 接口定位

当前后端围绕两个核心接口展开：

1. `POST /data-source/connect`
   - 用途：测试一个连接配置是否可用。
   - 特点：只返回成功/失败，不返回业务数据。
   - 典型场景：用户在前端填完连接信息后，点击“测试连接”。

2. `POST /data-source/execute`
   - 用途：基于指定连接执行一条 SQL。
   - 特点：查询语句返回 `rows`，写入语句返回 `affectedRows`。
   - 典型场景：用户选择某个连接后，在 SQL 编辑器中执行语句。

---

## 2. 总体设计思路

### 2.1 分层职责

当前采用比较清晰的三层职责：

- **Controller 层**
  - 只负责接收请求、调用 Service、返回统一结果。
  - 不写数据库连接细节，不直接拼 SQL。

- **Service 层**
  - 负责业务判断与流程控制。
  - 决定是测试连接还是执行 SQL。
  - 遇到错误直接抛出，由全局异常处理器统一返回失败结果。

- **ConnectionPoolManager 层**
  - 负责按连接配置创建和缓存连接池。
  - 相同连接配置命中同一个池，避免重复建连。
  - 程序退出时统一关闭所有池。

### 2.2 设计原则

- **统一入口**：前端只需要知道两个接口。
- **统一错误处理**：失败统一交给全局异常处理器。
- **资源复用**：外部数据库连接通过连接池复用，不长期持有单个 JDBC Connection。
- **可扩展**：后续可继续增加 Redis、MongoDB、图数据库等非 JDBC 类型。

---

## 3. 数据模型概览

### 3.1 连接请求模型

当前连接信息采用嵌套结构：

- 外层是通用字段：
  - `name`：用户自定义连接名称
  - `group`：分组信息
  - `dbType`：数据库类型
  - `config`：具体数据库配置对象

- 内层是多态配置对象：
  - `MysqlConnectionConfig`
  - `PostgreSqlConnectionConfig`

### 3.2 为什么这样设计

因为不同数据库的连接信息不完全一样：

- MySQL 这类 JDBC 数据库：通常只需要 `host / port / username / password`
- PostgreSQL：除了基础信息，还需要 `database`

所以把通用字段与数据库专属字段拆开，可以避免“所有数据库都强制带同一个字段”的问题。

## 4. 接口状态流转

## 4.1 连接测试 `/data-source/connect`

这个接口只做一件事：**验证连接是否可用**。

### 状态流转

```text
前端提交连接配置
  ↓
Controller 接收请求
  ↓
Service 校验 dbType / config.type
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行最小探活 SQL（SELECT 1）
  ↓
成功：返回 TulipeResult.success()
失败：抛出异常 → 全局异常处理器统一返回失败结果
```

### 状态解释

- **接收状态**：请求进入后端。
- **校验状态**：检查请求结构是否匹配当前支持的数据库类型。
- **建池/取池状态**：从缓存中复用已存在池，若不存在则创建。
- **探活状态**：执行 `SELECT 1` 验证数据库是否可连通。
- **结束状态**：
  - 成功：只返回“成功”标记。
  - 失败：统一输出失败信息。

### 成功后的含义

“连接成功”表示：

- 当前连接信息可用
- 认证通过
- 网络可达
- 数据库服务响应正常

它**不代表**后续 SQL 一定执行成功，因为后续语句还会受权限、表结构、SQL 内容影响。

---

## 4.2 SQL 执行 `/data-source/execute`

这个接口用于在指定连接上执行 SQL。

### 状态流转

```text
前端提交连接信息 + SQL
  ↓
Controller 接收请求
  ↓
Service 校验请求 SQL 是否为空
  ↓
Service 识别 SQL 类型（SELECT / INSERT / UPDATE / DELETE）
  ↓
ConnectionPoolManager 获取或创建连接池
  ↓
JdbcTemplate 执行 SQL
  ↓
返回 SqlExecutionResult
  ↓
失败：抛出异常 → 全局异常处理器统一返回失败结果
```

### SQL 类型处理

- `SELECT`
  - 返回查询结果集 `rows`
- `INSERT / UPDATE / DELETE`
  - 返回影响行数 `affectedRows`
- 其他类型
  - 当前先拒绝，避免首版过于复杂

### 为什么这样分

因为查询语句和写入语句天然不同：

- 查询语句的核心是“返回数据行”
- 写入语句的核心是“影响了多少行”

所以用一个统一结果对象承载，但内部字段按语义区分。

## 5. 连接池使用方式

### 5.1 连接池的目标

连接池不是“永远保持一个连接不放”，而是：

- 保持一个可复用的数据源对象
- 每次真正执行 SQL 时，从池里借出一个连接
- 用完后由连接池自动归还

### 5.2 当前实现方式

当前 `ConnectionPoolManager` 的行为是：

1. 根据连接配置生成一个缓存 key
2. 如果该 key 对应的池已存在，直接复用
3. 如果不存在，则创建新的 `HikariDataSource`
4. 程序退出时统一关闭所有池

### 5.3 为什么要这么做

好处有三个：

- **性能更好**：避免每次请求都重新建连
- **资源更稳**：连接会自动复用和回收
- **适合多连接场景**：每个用户连接都可以有自己的池

## 6. 返回结果说明

### 6.1 连接测试返回

`POST /data-source/connect`

- 成功：
  - 统一返回 `TulipeResult.success()`
  - `data = null`
- 失败：
  - 统一由异常处理器返回失败结果

### 6.2 SQL 执行返回

`POST /data-source/execute`

返回对象为 `SqlExecutionResult`：

- `sqlType`
  - SQL 类型
- `rows`
  - 仅查询语句使用
- `affectedRows`
  - 仅写入语句使用

### 6.3 这样设计的好处

- 接口返回结构统一
- 前端判断简单
- 后续可以继续扩展字段，比如：
  - `columns`
  - `costMs`
  - `truncated`
  - `warningMessage`

---

## 7. 当前支持范围与限制

### 7.1 当前支持

目前首版已支持：

- MySQL
- PostgreSQL
- JDBC 类型的连接池管理
- `SELECT / INSERT / UPDATE / DELETE` 的 SQL 执行

### 7.2 当前限制

- 非 JDBC 类型数据库暂未接入
  - 例如 Redis、图数据库、时序数据库等
- SQL 执行目前只开放常见单条语句
- 结果集暂未做分页与大结果裁剪
- 连接配置更新/删除后的池驱逐逻辑后续再补

---

## 8. 异常处理原则

当前采用统一异常处理策略：

- Service 层发现错误后直接抛出异常
- Controller 不吞异常
- 全局异常处理器负责转换成统一 `TulipeResult.failure(...)`

这样可以保证：

- 错误口径统一
- 前端处理简单
- 业务代码不被大量 `try/catch` 污染

---

## 9. 后续扩展建议

后续如果继续增强，建议按以下顺序推进：

1. 持久化连接信息到 SQLite
2. 加上连接列表、编辑、删除接口
3. 引入 `connectionId` 作为 SQL 执行主键
4. 为不同数据库抽象独立执行器
5. 接入 Redis、图数据库、时序数据库等非 JDBC 类型

---

## 10. 一句话总结

当前这套设计的核心目标是：

> **通过统一接口 + 连接池复用 + 服务层分流，把“测试连接”和“执行 SQL”做成可扩展、易理解、便于后续接 SQLite 持久化的基础能力。**
